# 4.5 Project Structure Overview — CampusCare

CampusCare is organized as a **monorepo** with two independent applications sharing the same repository:

```
CampusCare/
├── backend/       Express.js REST API (TypeScript + Prisma + PostgreSQL)
├── frontend/      React Native mobile app (Expo SDK 54)
└── docs/          Project documentation
```

---

## Backend Structure

```
backend/
├── prisma/
│   ├── schema.prisma      Database schema and ORM model definitions
│   ├── seed.ts            Script to populate the database with test accounts and sample data
│   └── setup.sql          Raw SQL reference used during initial Supabase setup
│
├── src/
│   ├── server.ts          Entry point — creates the Express app, registers all middleware and routes, starts the HTTP server
│   │
│   ├── config/
│   │   ├── db.ts          Exports the shared Prisma client instance used across all controllers
│   │   └── swagger.ts     Swagger/OpenAPI spec configuration for the interactive API docs at /api/docs
│   │
│   ├── controllers/       Business logic — one file per feature area
│   │   ├── auth.controller.ts          Register, login, logout, get current user (JWT issuance)
│   │   ├── issues.controller.ts        Create, list, get, update status, assign, close, comment, upload photo, delete
│   │   ├── notifications.controller.ts List notifications, mark all as read
│   │   └── admin.controller.ts         List all users, toggle active status, grant/revoke roles, list workers
│   │
│   ├── routes/            Express routers — map HTTP methods and URLs to controller functions
│   │   ├── auth.routes.ts             /api/auth/*
│   │   ├── issues.routes.ts           /api/issues/*
│   │   ├── notifications.routes.ts    /api/notifications/*
│   │   ├── manager.routes.ts          /api/manager/* (MANAGER + ADMIN only)
│   │   └── admin.routes.ts            /api/admin/* (ADMIN only)
│   │
│   ├── middleware/        Reusable Express middleware
│   │   ├── auth.ts        JWT verification (requireAuth) and role-based access guard (requireRole)
│   │   ├── error.ts       Global error handler — catches unhandled errors and returns consistent JSON error responses
│   │   └── upload.ts      Multer configuration for handling multipart photo uploads; saves files to uploads/
│   │
│   ├── services/
│   │   └── notify.ts      Helper function that creates Notification records in the database for a given user and issue
│   │
│   └── validators/
│       └── schemas.ts     Zod validation schemas for all request bodies (register, login, issue creation, status update, etc.)
│
├── uploads/               Runtime folder where uploaded issue photos are stored and served as static files
├── .env                   Environment variables (DATABASE_URL, JWT_SECRET, PORT — not committed to git)
├── .env.example           Template showing required environment variable keys
├── package.json           npm scripts and dependency declarations
└── tsconfig.json          TypeScript compiler configuration
```

### Key Files Explained

| File | Purpose |
|---|---|
| `src/server.ts` | Bootstraps the entire backend: applies security headers (Helmet), CORS, JSON parsing, rate limiting, mounts all routers, and starts listening on the configured port |
| `prisma/schema.prisma` | Single source of truth for the database structure — defines all models (User, Issue, Comment, etc.), enums (Role, IssueStatus, Priority, Category), and their relationships |
| `src/middleware/auth.ts` | Every protected route passes through `requireAuth`, which validates the JWT from the `Authorization` header and attaches the decoded user to `req.user`. `requireRole(...roles)` guards endpoints to specific roles |
| `src/validators/schemas.ts` | Zod schemas validate and sanitize all incoming request bodies before they reach controller logic, returning structured 400 errors on failure |
| `src/services/notify.ts` | Centralised notification writer — called by the issues controller whenever an issue status changes, is assigned, or is closed |

## Frontend Structure

```
frontend/
├── App.tsx                Root component — wraps the entire app in AuthProvider and renders AppNavigator
├── app.json               Expo configuration (app name, icon, splash screen, permissions)
├── eas.json               Expo Application Services config for cloud builds
│
├── assets/                Static assets (app icon, splash screen images)
│
└── src/
    ├── services/
    │   └── api.js         Central API layer — configures the axios instance, attaches JWT tokens via request interceptor,
    │                      exports named API objects (authAPI, issuesAPI, managerAPI, adminAPI, notificationsAPI),
    │                      and exports shared constants (CATEGORIES, PRIORITIES, STATUSES)
    │
    ├── utils/
    │   ├── AuthContext.js  React Context providing authentication state (user, token, role) and actions
    │   │                   (login, register, logout, refreshMe) to the entire component tree.
    │   │                   Also exports primaryRole() helper and ROLE_PRIORITY/ROLE_LABELS constants
    │   └── theme.js        Design system constants — color palette (COLORS) used across all screens and components
    │
    ├── navigation/
    │   └── AppNavigator.js Role-based navigation root — reads the authenticated user's primary role and renders
    │                       the appropriate navigator: AuthStack (guest), CommunityTabs (MEMBER),
    │                       WorkerTabs (WORKER), ManagerTabs (MANAGER), or AdminStack (ADMIN)
    │
    ├── components/
    │   └── UI.js          Shared reusable UI components:
    │                        • Button — primary / outline / secondary / danger variants
    │                        • StatusBadge — colour-coded badge for issue statuses
    │                        • PriorityBadge — colour-coded badge for issue priorities
    │                        • EmptyState — centered icon + message for empty lists
    │                        • LoadingScreen — full-screen activity indicator
    │
    └── screens/           All app screens, organized by user role
        │
        ├── auth/          Screens shown before login
        │   ├── LoginScreen.js       Email/password login form
        │   └── RegisterScreen.js    Registration form (name, email, password, phone)
        │
        ├── community/     Screens for MEMBER role (community members)
        │   ├── MyIssuesScreen.js    Lists all issues submitted by the logged-in member with status tracking
        │   ├── SubmitIssueScreen.js Form to submit a new issue (title, description, location, category, priority, optional photo)
        │   └── IssueDetailScreen.js Full detail view of a single submitted issue including photos, status, and comments
        │
        ├── worker/        Screens for WORKER role (maintenance workers)
        │   └── WorkerScreens.js    Contains two screens in one file:
        │                            • WorkerIssuesScreen — list of issues assigned to this worker
        │                            • WorkerIssueWorkScreen — work on a single issue: mark in-progress,
        │                              add completion notes, take a completion photo, mark as resolved
        │
        ├── manager/       Screens for MANAGER role (facility managers)
        │   ├── ManagerDashboardScreen.js   Overview of all issues with status stats, filter bar,
        │   │                               and the full scrollable issue list
        │   ├── ManagerIssueDetailScreen.js Manage a single issue: assign to worker, update status,
        │   │                               set priority, close or delete the issue
        │   └── WorkersScreen.js            List all registered workers; activate or deactivate accounts
        │
        ├── admin/         Screens for ADMIN role (system administrators)
        │   └── AdminUsersScreen.js  View all users, toggle account active status, grant or revoke any role
        │
        └── NotificationsScreen.js   Shared notifications screen (accessible from all role navigators):
                                     lists notifications, allows marking individual or all as read
```

### Key Files Explained

| File | Purpose |
|---|---|
| `App.tsx` | The single entry point rendered by Expo. Wraps everything in `SafeAreaProvider`, `StatusBar`, and `AuthProvider` before mounting `AppNavigator` |
| `src/services/api.js` | All HTTP calls go through this file. It exports `authAPI`, `issuesAPI`, `managerAPI`, `adminAPI`, and `notificationsAPI` — each grouping related endpoints. The axios interceptor automatically attaches the stored JWT to every request |
| `src/utils/AuthContext.js` | Provides the global auth state via React Context. Persists the JWT and user profile in `expo-secure-store` so the user stays logged in across app restarts. Exposes the `useAuth()` hook used in every screen |
| `src/navigation/AppNavigator.js` | The entire navigation tree lives here — four separate navigator stacks/tabs (one per role) are defined and switched based on `primaryRole(user)` |
| `src/components/UI.js` | Centralises all common visual elements so styles and behaviour are consistent across all role-specific screens |

## Data Flow Summary

```
Mobile App (React Native / Expo)
        │
        │  HTTP/JSON  (axios + Bearer JWT)
        ▼
Express API Server  (Node.js / TypeScript)
   ├── Middleware:  auth.ts → validates JWT, attaches req.user
   ├── Routes:      match URL → call controller function
   ├── Controllers: validate input (Zod) → query database (Prisma) → return JSON
   └── Services:    notify.ts writes Notification records as side effects
        │
        │  Prisma ORM (connection pooling via PgBouncer)
        ▼
PostgreSQL Database  (Supabase hosted)
```
